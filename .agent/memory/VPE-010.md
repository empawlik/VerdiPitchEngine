# Technical Resolution: VPE-010 - Batch Architecture & Testing Mocks

## Architectural Decision
Due to the constraints of the QNAP ecosystem and remote execution, processing massive lossless audio structures interactively required a robust batching mechanism. A sequential orchestration script (`verdi-batch`) was chosen over Go-native concurrent directory climbing to prevent CPU/Memory thrashing when queuing potentially hundreds of high-resolution albums simultaneously.

## Implementation Details
- **Mocking Exec Command:** The Go `os/exec` package limits the ability to inject fake system commands. To bypass this and push `internal/converter` coverage over 80%, we mapped `exec.Command` and `exec.CommandContext` to package-level variables. This allowed the test suite to hijack the invocation, launching a pseudo-process (`TestHelperProcess`) that fakes `ffmpeg` standard output without requiring a local `librubberband` installation.
- **Atomic Renaming:** Output files are explicitly given a `.tmp` extension during the conversion phase. They are atomically renamed to `.flac` only upon successful completion. This protects against partial media streams being ingested by NAS media scanners (e.g., Plex, Roon) before `ffmpeg` has fully written the headers.
- **Terminal Decorators:** The `mpb` multi-progress bar caused terminal deadlocks across container execution contexts when pushing high-frequency TTY updates. Extraneous decorators were stripped to guarantee script idempotency and headless stability.
