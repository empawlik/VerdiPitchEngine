# Explorative Task Post-Mortem: VPE-002

> [!ABSTRACT] Core Concept
> Architectural review and future-scoping of the VerdiPitchEngine end-to-end FFmpeg integration tests (VPE-002), identifying limitations in local test parity.

## 1. Current State Evaluation
The `VPE-002` implementation successfully established an End-to-End integration test (`TestE2EFFmpegPitchShift`) that natively executes the `ProcessFile` pipeline on a dynamically generated 440 Hz synthetic FLAC tone. We enforce strict zero-trust principles by locking all intermediate file writes to `t.TempDir()`. The implementation successfully gracefully handles missing dependencies via the `hasRubberband()` guardrail, which skips the test on standard macOS local environments missing the specialized library while successfully executing on NAS/containerized runners. The package currently sits at a highly healthy 90.6% statement coverage.

## 2. Pros and Cons
**Pros:**
- Complete end-to-end integration proving the orchestration works as expected without requiring actual user master files.
- `hasRubberband()` ensures developer quality-of-life remains high, preventing local `mage test` suites from failing erroneously due to missing OS packages.
- I/O remains strictly within the ephemeral testing bounds, preventing storage pollution.

**Cons:**
- The actual success criteria of the test is relatively shallow; `ffprobe` checks for standard file integrity, but the test does not explicitly analyze the frequency spectrum to mathematically prove the 440 Hz -> 432 Hz shift actually occurred.
- Relying on conditional skipping means the most critical functional test of the application is implicitly disabled for many local developers.

## 3. Alternative Solutions
We considered embedding a small pre-rendered 440 Hz FLAC fixture inside the repository to avoid synthesizing audio on the fly. This was discarded to avoid repository bloat over time and strictly adhere to zero-trust data constraints where tests generate their own independent state cleanly.

## 4. Brainstorming / Future Enhancements
1. **[VPE-005] Containerized E2E Runner**: Introduce a `mage test:docker` target that automatically bundles the test binary and executes the skipped tests inside a localized replica of the NAS container, ensuring developers can run the integration tests explicitly without modifying their host macOS FFmpeg installations.
2. **[VPE-006] Spectral Analysis Validation**: Extend the E2E verification logic to parse the output spectrum (via Go DSP libraries or a specialized FFmpeg `showfreqs` probe) to mathematically assert that the fundamental frequency of the output FLAC shifted explicitly to 432 Hz.

## 5. ATLAS Documentation Usage
The task operated on the `Go` and `FFmpeg` tech stack. During the initial `task-validate` phase, the repository labels `atlas:go` and `atlas:ffmpeg` were applied to the remote tracker. Local knowledge and established OS/exec patterns from `internal/converter/ffmpeg.go` were explicitly referenced. No additional remote ATLAS markdown schemas needed to be queried directly, as the architectural constraints (NAS parity, `librubberband`) were well understood from the previous post-mortem.
